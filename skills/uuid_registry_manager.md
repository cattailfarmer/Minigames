# Skill: uuid_registry_manager
name: uuid_registry_manager
description: Central, authoritative registry for all UUIDs generated by the specification/justification/solution system. Records what each UUID represents, where it resides, and all edges (parent/child, justification, dead-end links). Prevents UUID explosion by enforcing single-source identity management.
case: Summoned automatically by specification_generator, justification_and_solution_generator, or any other skill that creates or references a UUID. Always operates under delineation and epistemic-uncertainty.
file: uuid_registry_manager.md
dependencies: delineation, epistemic-uncertainty

## Purpose (Waist)
The UUID Registry Manager is the single source of truth for identity in the specification system. It ensures every UUID is registered at the moment of creation, with full metadata and edge relationships, so the entire graph remains traceable and queryable.

## Workflow (3-Boundary Pants Structure)

### Waist – Registry Declaration
Load or initialize the central registry file (uuid_registry.yaml or harnessed store).

### Leg 1 – UUID Registration
For every new UUID generated:
- Record: UUID, type (requirement, solution_element, justification_node, dead_end, etc.), parent_UUID (if any), file_path, section_path, specification_version, timestamp, generating_skill.
- Validate that the UUID is truly unique before accepting it.

### Leg 2 – Edge & Relationship Logging
- Register all edges immediately (justified_by, prerequisite_for, dead_end_of, etc.).
- If a dead-end is marked, link it explicitly so future calls can query and avoid repeating the path.

Output confirmation to the calling skill and update the registry atomically.

## Core Rules (always enforce)
- No UUID may be emitted by any skill until it has been successfully registered.
- Every edge must be recorded at creation time.
- Maintain versioned history of the registry itself so recompilation can trace lineage.
- Stay under epistemic-uncertainty to flag any orphaned UUIDs or broken edges.

## Integration Rules
- Automatically called by specification_generator and justification_and_solution_generator before any UUID is emitted.
- Can be queried by Observer, Planner, or any Faculty needing traceability.
- Feeds directly into recompilation cycles and harness memory.

## Boundaries / When Not to Use
- Do not use for non-UUID identifiers.
- Do not bypass registration for performance reasons.

## Resolution
Every UUID and every edge is now centrally registered, eliminating explosion and providing full traceability across the entire specification/justification/solution graph.
